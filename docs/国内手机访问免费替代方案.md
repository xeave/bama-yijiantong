# 国内手机访问免费替代方案

## 1. 先说结论
当前已经切到 GitHub Pages 路线，目标地址是 `https://xeave.github.io/bama-yijiantong/`。

如果在中国大陆手机网络下仍然打不开，问题大概率不在页面代码，而在免费海外托管的访问链路本身。

这类纯静态站点可以直接迁移，不需要改业务逻辑。

## 2. 现在最适合的三个方案
### 方案 A：GitHub Pages
适合：
- 想要真正 0 成本
- 先快速换一个地址试试
- 不想买域名

优点：
- GitHub 官方免费
- 当前项目是纯静态站，基本可以直接发
- 代码已经是相对路径写法，兼容这类静态托管

缺点：
- 在中国大陆手机网络下不保证一定稳定
- 只是比 `vercel.app` 多一个免费试错选择

适合当前阶段的原因：
- 完全免费
- 最快
- 不需要备案

补充提醒：
- 如果你现在用的是 `GitHub Free`，通常需要把仓库设为 `Public` 才能用项目级 Pages

### 方案 B：腾讯云 CloudBase 静态网站托管
适合：
- 主要用户在中国
- 想先验证手机访问是否明显改善
- 可以接受“新用户免费额度”而不是永久免费

优点：
- 腾讯云官方静态托管
- 自带 HTTPS
- 支持默认域名
- 支持直接上传文件夹或 ZIP

缺点：
- 不是永久完全免费
- 官方当前写的是新用户有一个月免费额度，之后按量计费

适合当前阶段的原因：
- 你现在最在意的是中国手机能不能打开
- 这个方向比继续用海外默认域名更值得试

### 方案 C：EdgeOne Pages + 自定义域名
适合：
- 不想买服务器
- 可以只花一个域名钱
- 想继续走静态站 + CDN 的轻量路线

优点：
- 托管本身可以免费试
- 静态项目迁移成本低
- 更适合后续正式对外传播

缺点：
- 如果选择“全球可用区（不含中国大陆）”，中国大陆访问项目默认域名会返回 `401`
- 想稳定给中国手机访问，官方建议绑定自定义域名
- 自定义域名本身通常要花钱

适合当前阶段的原因：
- 如果你愿意只承担域名成本，这是后面更像正式站点的路线

## 3. 我更推荐的顺序
如果你现在目标是“先免费再试一次”，建议按这个顺序：

1. 先试 `GitHub Pages`
2. 如果国内手机还是不理想，再试 `腾讯云 CloudBase`
3. 如果确认要长期做，再上 `EdgeOne + 自定义域名`

原因很简单：
- 第一步完全 0 成本
- 第二步更偏国内访问优化
- 第三步更适合准备长期传播

## 4. GitHub Pages 最短操作
这个项目本身已经可以直接用于 GitHub Pages。

操作步骤：

1. 把项目推送到 GitHub 仓库 `bama-yijiantong`
2. 如果你使用的是 `GitHub Free`，确认仓库为 `Public`
3. 进入 `Settings`
4. 打开 `Pages`
5. 在 `Build and deployment` 中确认 `Source` 为 `GitHub Actions`
6. 打开 `Actions`
7. 等待 `Deploy GitHub Pages` 发布完成

如果 `Pages` 入口不可用，先检查：
- 当前仓库是不是 `Public`
- 当前账号是不是免费版且仓库仍为私有

发布后通常会得到类似地址：

`https://xeave.github.io/bama-yijiantong/`

然后重点用中国手机网络测试：
- 首页能不能打开
- `meds/`、`health/`、`calculator/`、`nav/` 能不能正常跳转
- `localStorage` 保存是否正常

## 5. 腾讯云 CloudBase 最短操作
如果你想更偏向中国访问环境，可以直接试这个。

操作步骤：

1. 登录腾讯云
2. 打开“云开发 CloudBase”
3. 开通环境
4. 进入“静态网站托管”
5. 选择“上传代码包/文件夹”
6. 直接上传当前项目根目录，或者上传 ZIP
7. 等待部署完成
8. 用系统分配的默认域名在手机里测试

建议先不要接自定义域名，先只看：
- 打不开的问题有没有改善
- 打开速度有没有明显提升

## 6. EdgeOne 要特别注意的点
如果你准备用 EdgeOne Pages：

- 想免备案，就要选“全球可用区（不含中国大陆）”
- 但这个模式下，中国大陆访问项目默认域名会返回 `401`
- 官方建议绑定自定义域名来建立稳定访问通道

所以它更像是：
- 托管免费
- 域名可能要花一点钱

而不是彻底零成本。

## 7. 当前项目是否需要改代码
目前不需要。

原因：
- 首页和子页面都用了相对路径
- 纯 HTML / CSS / JS
- 没有后端依赖
- 没有构建流程要求

也就是说，这个项目已经适合直接迁移到其他静态托管平台。
